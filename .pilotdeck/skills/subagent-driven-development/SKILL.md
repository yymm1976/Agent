# Subagent-Driven Development

每个任务派一个独立子 Agent，Phase 完成后交叉审查两遍。隔离上下文保证专注。

**核心原则：** 从 Plan 一次性提取所有任务 → 每个子 Agent 干一个任务 → Phase 完成后交叉审查 → 下一个任务。除非被阻塞，任务之间不停顿。

## The Process

1. 读一次 Plan 文件，把所有任务的完整文本提取到 TodoWrite
2. 对每个任务：
   - 按「父 Agent 给子 Agent 的 Prompt 规范」组装 prompt
   - 派发实现子 Agent
   - 子 Agent 实现、构建、提交
   - 标记任务完成
3. 所有任务完成后：对整体实现做 Phase 级交叉审查（两轮）

## 状态处理

| 状态 | 动作 |
|------|------|
| DONE | 标记完成，继续下一个任务 |
| DONE_WITH_CONCERNS | 先读 CONCERN，FATAL 立即处理，RISK 记录继续 |
| NEEDS_CONTEXT | 补充缺失信息，重新派发 |
| BLOCKED | 评估：补上下文 / 拆分任务 / 上报人类 |

### 升级处理（统一出口）

以下情况统一进入「升级处理」流程，不再分散描述：
- 同一错误出现 2 次 → 标记 RISK，改变策略（换思路 / 拆更细 / 补充上下文）
- 子 Agent 连续构建失败 3 次（含不同错误）
- 子 Agent 报告 BLOCKED 且父 Agent 无法自行解决
- 子 Agent 报告 FATAL 级 CONCERN

**升级处理步骤**：
1. 父 Agent 收集失败信息（错误输出、CONCERN 描述、当前状态）
2. 判断根因：环境问题 → 调 env-doctor；代码问题 → 重新派发或父 Agent 自己修；架构问题 → 暂停 Phase，上报人类
3. 修复后重新派发，从失败点继续（不从头开始）

## 父 Agent 给子 Agent 的 Prompt 规范（核心）

子 Agent 是独立会话，只加载 `user_rules.md`（全局规则），不加载记忆、不加载 Skill、不继承父 Agent 历史。所以父 Agent 必须在 prompt 中提供子 Agent 需要的一切。

### 必须包含（3 件事）

**1. 任务描述**
- 来自 Plan 的具体任务文本
- 只描述"做什么"，不解释"为什么"

**2. 相关规则片段（渐进式披露）**
- 只放和当前任务相关的规则，不涉及的不放
- 子 Agent 已加载全局规则（铁律/命令/优先级/Phase/交付），不需要重复
- **原则**：按任务类型只放相关规则片段，不全量粘贴。具体哪些规则对应哪些任务类型，由父 Agent 根据项目领域知识判断（Minecraft 模组项目的规则映射见 `user_profile.md` 的各领域章节）
- **踩坑清单**：有 Cross-Project Pitfalls 的任务类型（实体类、容器类等），务必附带相关条目（见 `user_profile.md` Task-Rule Mapping），否则踩坑清单对子 Agent 静默失效

**3. 上下文定位**
- 相关类的全限定名和文件路径
- 需要直接改动的关键片段（仅必要的几行）
- 其余让子 Agent 用项目可用的代码索引工具取

### 禁止包含

- ❌ 完整的行为规范模板（子 Agent 已加载全局规则）
- ❌ 完整的优先级列表（只放相关的）
- ❌ 解释性文字（"为什么"）
- ❌ 跨项目易错点（子 Agent 不需要）
- ❌ 工具链映射表（子 Agent 已知道）

> 交付格式模板随共享上下文下发，不算重复。

### Prompt 模板

```
## 任务
[具体任务描述，来自 Plan]

## 相关规则
[只放和当前任务相关的规则片段，1-3 条]

## 上下文
- [全限定名]，路径 [文件路径]，作用 [一句话]
- ...

需要直接改动/依赖的关键片段（仅这些，其余用代码索引工具取）：
[只贴必要的几行代码]

其余用代码索引工具按需取，禁止通读整个项目。
```

### 示例

**好：**
```
## 任务
在 UserModule 中注册 UserService，实现基础用户查询行为。

## 相关规则
- 注册用 DI 容器，注意在正确的生命周期阶段注册
- 配套项：接口定义、配置文件、单元测试

## 上下文
- com.example.app.module.UserModule, src/main/java/.../module/UserModule.java, 用户模块注册中心

需要改动的关键片段：
[DI 容器声明 3 行]
[一个注册示例 2 行]

其余用代码索引工具按需取，禁止通读整个项目。
```

**坏：**
```
## 任务
在 UserModule 中注册 UserService。

## 行为规范（❌ 子 Agent 已加载全局规则，不要重复）
你是项目执行人。你的全部价值在于...

## 完整优先级（❌ 只放相关的）
1. 版本与 API 正确性...
2. 框架适配...

## 完整交付格式（❌ 子 Agent 知道格式）
Phase [N] 执行报告...
```

## Token 优化：精准上下文，不要全量粘贴

**这是本 Skill 最费 token 的地方，务必遵守。**

❌ **旧做法（费 token）**：父 Agent 把主类、所有注册类、项目主配置文件的完整代码粘进每个子 Agent 的 prompt。

✅ **新做法（省 token，交付质量不变）**：

1. **父 Agent 不预读全量文件**，改为给子 Agent 提供「定位信息 + 关键片段」
2. **子 Agent 的取数纪律**：优先用代码索引工具获取上下文，禁止通读整个项目
3. **共享上下文一次性给全**：项目通用信息（项目标识符、包根路径、框架版本等），在第一个子任务里讲清楚，后续任务只需引用
4. **交付格式随共享上下文下发**：子 Agent 只知道格式名称不知道具体字段，在第一个子任务的共享上下文里附带报告格式模板

## 子 Agent 行为规范（精简版，写进 prompt 尾部）

派发实现子 Agent 时，在 prompt 尾部追加以下精简规范：

```
## 执行纪律
- 构建失败最多修复 2 次，第 3 次失败返回父 Agent 处理
- 每次回合最多向父 Agent 提 1 个问题，超出则合并到下一次提问
- 禁止向父 Agent 询问可通过代码索引工具或项目文件自行获取的信息

## 自检
执行前列出：本任务中我无法独立完成的事项（如有）。无则写「无」。
```

> 只追加全局规则未覆盖的执行纪律。铁律/CONCERN/构建验证等子 Agent 已加载，不重复。

---

## Phase 级交叉审查（替代 Task 级审查）

Task 级别不再派独立审查子 Agent（Task 已切得很碎，审查收益低且费 token）。改为 Phase 完成后做两轮交叉审查：

**第一轮：架构/耦合审查**
- 跨模块调用是否正确（用代码索引工具验证调用关系）
- 注册/初始化一致性
- 数据持久化完整性
- 框架特定安全约束（如 Side 隔离、线程安全）

**第二轮：细节/质量审查**
- 命名一致性、边界条件、魔法数字
- 代码注释质量
- 生成资源与注册内容一致

**审查方式**：派一个审查子 Agent，prompt 包含：
- 整个 Phase 的变更 diff（用 `rtk git diff` 取得，直接粘入）
- 审查标准——将 code-reviewer 五维清单的关键条目文本**直接粘入 prompt**（子 Agent 无法访问 Skill 文件，传路径无效）
- 范围声明（只审这些代码）

两轮审查可以合并为一次派发（一个子 Agent 做两轮），也可以分两次派发（更彻底但更费 token）。

---

## 审查后处理

审查子 Agent 按 Critical / Important / Minor 三级返回：

- **Critical**：立即修复，阻塞后续
- **Important**：修后继续
- **Minor**：记录到 CONCERN 索引，后续处理

若你认为 reviewer 判断有误：用技术理由反驳，给出能证明代码正确的依据。

---

## Rules

- 禁止并行派发多个实现子 Agent（会冲突）
- 禁止让子 Agent 读 Plan 文件——直接给它任务完整文本
- 子 Agent 提问后，先回答再让它继续
- 连续失败或 BLOCKED：进入「升级处理」流程，不自行硬闯

## 不适用于

- 任务高度串行依赖（后一个任务必须等前一个的输出，子 Agent 隔离反而增加沟通成本）
- Plan 颗粒度太粗（一个任务包含多个不相关改动，子 Agent 无法聚焦）
- 没有可靠构建验证（无法自动判断完成与否，子 Agent 会反复失败）
