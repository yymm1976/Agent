# RouteDev Tool/Skill Description 书写规范

> **核心原则：description 是写给模型看的，不是写给人看的。它决定了 80% 的工具/Skill 匹配效果。**
>
> 模型在每一次决策时，会根据用户意图与所有可用工具/Skill 的 description 做语义匹配。
> description 写得越精准、越能体现「何时该用」，模型的选择就越准确；
> 反之，模糊、冗长、缺少触发场景的 description 会导致误匹配、漏匹配，进而拖垮整条 Agent 链路。

本规范适用于：
- `src/tools/builtin/*.ts` 中 `ToolDefinition.description` 字段
- 任意 `SKILL.md` 文件 frontmatter 中的 `description` 字段
- 通过插件/MCP 注册的自定义工具描述

---

## 1. 句式模板

**推荐句式：**

```
当用户 [触发场景] 时，使用此 [工具/Skill]。[适用条件/边界]。
```

英文等价：

```
Use this [tool/skill] when [trigger scenario]. [conditions/constraints].
```

### 模板拆解

| 组成 | 作用 | 示例 |
|------|------|------|
| 触发场景 | 描述「何时该用」，是模型匹配的关键 | 需要查看某个文件的内容 |
| 工具/Skill | 明确主体，避免歧义 | 此工具 |
| 适用条件 | 补充边界、限制、能力范围 | 支持指定行号范围与文件大小限制（1MB） |

### 为什么是这个句式

- 「当用户……时」明确触发条件，模型可基于用户意图直接命中
- 「使用此工具」让模型在多工具候选时清晰区分
- 末尾的适用条件用于排除误用（如大小限制、权限要求、回退策略）

## 2. 差/好示例对比

### 示例 A：file_read

❌ **差：**
```
读取文件内容。支持指定行号范围。返回文件的文本内容。文件超过 1MB 会拒绝读取，请用 startLine/endLine 分段读取。
```
问题：没有触发场景，模型不知道「什么时候」该选它；罗列实现细节，喧宾夺主。

✅ **好：**
```
当用户需要查看某个文件的内容、理解现有代码实现、或在修改前确认当前代码时，使用此工具。支持指定行号范围与文件大小限制（1MB）。
```

### 示例 B：shell_exec

❌ **差：**
```
执行 Shell 命令。返回命令的标准输出和标准错误。超时时间由 context 控制。
```
问题：仅描述「能做什么」，没有说明「何时该用」，与 file_search 等工具难以区分。

✅ **好：**
```
当用户需要执行 shell 命令、运行构建脚本、或调用系统工具时，使用此工具。执行前会进行环境变量白名单过滤与熔断保护。
```

### 示例 C：web_search

❌ **差：**
```
搜索网页。
```
问题：过短，缺少触发场景与能力边界，模型无法判断与 web_fetch 的差异。

✅ **好：**
```
当用户需要搜索网页获取最新信息时，使用此工具。多引擎回退（11 引擎），优先中国直连引擎，返回标题、链接和摘要。
```

### 示例 D：spawn_agent

❌ **差：**
```
生成子 Agent。
```
问题：仅 6 个字，模型完全无法判断何时该派子 Agent。

✅ **好：**
```
当用户需要并行处理多个独立子任务、隔离上下文提高专注度时，使用此工具生成子 Agent。子 Agent 有独立的迭代空间，结果返回给主 Agent。
```

## 3. 检查清单

在提交工具/Skill 前，逐项核对：

- [ ] **长度 ≥ 20 字符**：过短的 description 无法承载触发场景
- [ ] **包含触发场景**：使用「当用户……时」或等价英文「when ...」开头
- [ ] **包含动词**：描述中至少出现一个动作动词（查看/创建/搜索/执行/抓取/管理/记录 等）
- [ ] **包含触发关键词**：至少含 `当`/`需要`/`时`/`使用` 之一（中文）或 `when`/`need`/`use` 之一（英文）
- [ ] **避免纯实现描述**：不把「返回 stdout/stderr」「超时由 context 控制」等实现细节作为主体
- [ ] **与同类工具有区分度**：file_search vs code_search vs repo_map 必须能从 description 区分
- [ ] **不超过 200 字符**：过长会稀释关键信息，模型匹配权重下降
- [ ] **不包含工具名自身**：模型已知 name 字段，无需在 description 中重复
- [ ] **明确边界与限制**：如大小限制、权限要求、回退策略等
- [ ] **避免营销话术**：不写「强大的」「高效的」「最佳」等主观词

## 4. 自动审计

项目提供 `scripts/lint-descriptions.ts` 脚本，自动扫描：

1. `src/tools/builtin/*.ts` 中的 `description:` 字段
2. 任意 `SKILL.md` 文件 frontmatter 中的 `description` 字段

### 规则

| 规则 | 说明 | 严重级别 |
|------|------|----------|
| MIN_LENGTH | 长度 < 20 字符 | error |
| NO_TRIGGER | 缺少触发关键词（当/需要/时/使用/when/need/use） | error |
| NO_VERB | 缺少动作动词 | warning |

### 运行

```bash
pnpm tsx scripts/lint-descriptions.ts
```

退出码：0 = 全部通过，1 = 有 error。

### 接入 verify.ts

`scripts/verify.ts` 已集成 description lint 检查项（过渡期以 warning 形式报告，不阻断验收）。
