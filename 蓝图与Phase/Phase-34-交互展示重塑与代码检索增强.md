# Phase 34 — 交互展示重塑与代码检索增强

> **版本目标：** v2.6.0
> **前置依赖：** Phase 33 完成（v2.5.0）
> **新增测试要求：** ≥ 20 个
> **研究依据：** 《Agent 工具交互展示方案研究报告》+ 《实用型 Coding Agent 功能体系解构与自研项目构建指南》
> **核心命题：** RouteDev 的"信息密度控制权"在开发者手里（DisclosureLevel 配置），但不在用户手里。轻度用户看到太多过程信息会焦虑，重度用户又觉得信息不够。本 Phase 的目标是把这个控制权交给用户，同时补上代码检索和过程评测两个架构短板。

---

## 研究背景：业界在怎么做

> 以下内容来自两份深度研究报告的交叉验证。不是标准答案，是思考素材。

### 竞品共性与差异

报告调研了 Codex CLI、Claude Code、Cursor 三款产品的工具交互展示方式，发现了三个共识：

1. **信息减法而非加法**——所有产品默认展示越少越好。Codex CLI 只展示"过去时动词 + 结果"（如 `Applied`、`Wrote`），Claude Code 只展示"一行摘要"，Cursor 用步骤列表但也默认折叠详情。
2. **用户控制权**——Claude Code 的 `/output-style` 命令最值得借鉴：它让用户（而非开发者）决定信息密度。这是"兼顾轻度/重度用户"的最佳实践。
3. **不依赖复杂动效**——Codex 和 Claude Code 在终端里都不用 Spinner 动画系统。但报告同时指出：如果目标用户包括非专业开发者，完全无动效会导致"卡死焦虑"。

### 关键差异：Codex vs Claude Code

| 维度 | Codex CLI | Claude Code |
|------|-----------|-------------|
| 思考过程 | 完全隐藏 | 默认折叠，可展开 |
| 工具调用 | 过去时动词 + 结果 | 一行摘要 + 状态 |
| 完成后 | 变更摘要 | 保持原样 |
| 失败时 | 错误信息 | 自动展开 thought |
| 用户控制 | 沙盒模式决定密度 | `/output-style` 命令 |

**对 RouteDev 的启示：** 我们同时有 CLI（Ink 终端）和 Desktop（Electron + React）两套界面。Desktop 的信息密度天然更高（工具卡片、消息气泡、侧边栏面板），如果照搬 Codex 的极简模式，Desktop 用户会觉得"界面太空了"。但如果照搬 Cursor 的高密度模式，轻度用户会被淹没。正确的做法是：**一套配置，两套渲染**——outputStyle 控制信息密度，CLI 和 Desktop 各自按自己的 UI 范式实现。

## Task 1：Output Style 系统——让用户控制信息密度（6 测试）

### 1.1 问题定义

当前 RouteDev 用 `ui.disclosureLevel`（1=摘要/2=关键细节/3=完整数据）控制信息披露。这个设计方向是对的，但报告提出了一个更精准的模型——**Output Style**，它不只控制"展示多少"，还控制"怎么展示"：

| 维度 | minimal（轻度用户） | standard（普通用户，默认） | verbose（重度用户） |
|------|:---:|:---:|:---:|
| 思考过程 | 完全隐藏 | 折叠但可展开 | 默认展开 |
| 工具调用 | 仅展示结果（过去时动词） | 一行摘要 + 状态图标 | 完整参数 + 返回结果 |
| 进度信息 | 仅当前步骤目标 | 步骤计数 + 进度条 | 完整时间轴 + Token 消耗 |
| 完成后行为 | 立即折叠，只留结果 | 折叠 + 微摘要 | 保留完整过程 |
| 动效 | 无 | Spinner + 计时器（>30s） | 全部展示 |

### 1.2 架构建议

**不是替换 `disclosureLevel`，而是升级为 `outputStyle`。** `disclosureLevel` 是一个数字（1/2/3），`outputStyle` 是一个枚举（`minimal`/`standard`/`verbose`）。语义更清晰，扩展性更好。

需要思考的问题：
- `disclosureLevel` 和 `outputStyle` 是否可以共存？还是一步到位替换？如果替换，需要处理现有配置文件的向后兼容。
- `outputStyle` 应该放在 `ui` section 还是新的 section？报告建议放在 `ui` section。
- 是否需要运行时切换命令（如 `/output-style minimal`）？Claude Code 有这个命令，体验很好。

### 1.3 CLI 端实现要点

Ink 终端的三层信息架构：

- **L1（始终可见）**：StatusBar 组件，展示模型/模式/当前动作——已经存在，根据 outputStyle 增减内容。
- **L2（standard/verbose 可见）**：StepCard 组件，展示动作流——已存在，需要加条件渲染。`minimal` 模式下隐藏非当前步骤。
- **L3（verbose 可见）**：TracePanel 组件，展示完整 span 日志——已存在，需要控制默认展开层级。

键盘交互：按 `d` 键在 L1→L2→L3 之间切换（TracePanel 已实现此功能），让用户在运行时临时查看更详细的信息，无需改配置。

### 1.4 Desktop 端实现要点

Electron 的 ChatPage 已有工具卡片（ToolCard）和消息气泡。Output Style 的映射方式需要你思考：

- `minimal`：工具卡片折叠为单行摘要（"已修改 utils.py +12/-3"），点击展开
- `standard`：工具卡片显示摘要 + 状态图标，可展开查看详情
- `verbose`：工具卡片默认展开，显示完整参数和返回结果

Desktop 有 Cursor 没有的优势：**可以用鼠标点击展开/折叠**，不需要键盘操作。这比 CLI 的 `d` 键切换更直觉。

### 1.5 思考引导

报告的核心发现是"没有工具使用复杂的动画或 Spinner 系统"。但这不意味着 RouteDev 也不该用——竞品面向的是专业开发者，对"终端卡住"有心理预期。如果你的用户包括轻度用户，适当的动效是必要的。

**需要你决定的：** RouteDev 的目标用户画像是什么？如果主要是开发者，可以大胆做减法；如果包括产品经理、测试人员等非技术用户，需要保留更多反馈信号。这个判断会影响 outputStyle 各层级的具体渲染行为。

## Task 2：完成后折叠与微摘要系统（5 测试）

### 2.1 核心设计

报告提出了一个**非对称折叠策略**——成功和失败的展示行为不同：

```
✅ 成功 → 折叠过程 + 展示微摘要
❌ 失败 → 自动展开过程 + 高亮错误
⏸️ 暂停 → 不折叠，等待用户决策
```

这个不对称设计的原因很直觉：成功时用户只需要结果，过程是噪音；失败时用户需要诊断信息，过程是线索。

### 2.2 微摘要的内容

报告建议微摘要包含四个要素：

```
✅ 任务完成：utils.py 性能优化
─────────────────────────────
📊 摘要：4 步 / 2m15s / +12 -3 / 测试通过

💡 关键决策：
  • 将 O(n²) 循环改为 Map 查找（性能提升 ~10x）

[展开执行过程]  [查看 Diff]
─────────────────────────────
```

其中"关键决策"的提取方式需要思考：

- **方案 A**：在系统提示词中要求 LLM 用 `<decision>` 标签标记关键决策，前端解析后展示。这需要修改提示词模板，但提取精准。
- **方案 B**：从 ReActEvent 流中自动提取——比如检测"模型选择了方案 X 而非方案 Y"的转折点。不需要改提示词，但提取可能不准确。
- **方案 C**：不提取关键决策，微摘要只包含统计信息（步骤数/耗时/文件变更数）。最简单，但信息价值最低。

**需要你决定的：** 方案 A 需要在 `main.system` 提示词模板中增加一段指令，告诉 LLM 在完成任务时用 `<decision>` 标签。这会影响所有模型的输出行为——有些模型可能不遵守标签格式，有些模型可能生成太多 `<decision>` 导致微摘要过长。你需要评估这个 trade-off。

### 2.3 Desktop 端的实现

在 ChatPage 中，"完成后折叠"意味着：Agent 的整段执行过程（多个消息气泡 + 多个工具卡片）在完成后被收入一个"任务完成卡片"中。用户点击可展开查看完整历史。

这需要和现有的消息流渲染逻辑配合——不是一个新组件，而是在消息列表的渲染逻辑中加一个"折叠组"的概念。

**需要你思考的：** 在 CLI 端，折叠意味着"用更少的行展示"。在 Desktop 端，折叠意味着"把多个消息气泡收进一个卡片"。两端的"折叠"语义不同，但 outputStyle 配置是同一个。如何保证用户在两端切换时体验一致？

## Task 3：动作动词体系与工具执行反馈（4 测试）

### 3.1 动词过去时风格

报告发现 Codex CLI 使用"过去时动词"（`Applied`、`Wrote`、`Patched`）展示已完成的工具调用，而非进行时（`Applying...`、`Writing...`）。这个设计选择的原因：过去时强调"已完成"，给用户确定感；进行时强调"正在做"，增加焦虑。

建议 RouteDev 建立类似的动词体系，区分执行中/执行后/失败三种状态：

| 工具类型 | 执行中 | 执行后 | 失败 |
|---------|--------|--------|------|
| 读取文件 | `读取 config.json...` | `已读取 config.json (4KB)` | `读取失败：权限不足` |
| 编辑文件 | `修改 utils.py...` | `已修改 utils.py (+12/-3)` | `修改失败：匹配失败` |
| 执行命令 | `运行 npm test...` | `测试通过 (3/3)` | `测试失败 (2/3)` |
| 搜索代码 | `搜索认证逻辑...` | `找到 5 个匹配` | `搜索无结果` |

### 3.2 极简动效

报告的核心判断：**只在工具执行阶段使用动效，LLM 推理阶段不需要动效。**

原因：LLM 推理时已有流式 token 输出，用户能看到"在思考"。工具执行时没有流式输出，用户看到的是"空白等待"——这才是焦虑的来源。

建议保留两个动效元素：
1. **工具执行时的简洁 Spinner**（`⠋` 单字符旋转，每 200ms 切换）——CLI 端用 Ink 的 `<Spinner>` 组件，Desktop 端用 CSS 动画
2. **超过 30 秒时显示计时器**（"已运行 45s"）——消除"等了多久"的不确定性

### 3.3 超时与取消

报告建议超过 90 秒时显示取消提示。这对应 RouteDev 已有的 `confirmTimeout` 机制，但需要确认：当前的超时行为是什么？是自动取消还是等待？对于 Desktop 端，90 秒后应该显示一个"取消执行"按钮。

**需要你决定的：** Spinner 的帧率。报告建议 200ms 切换（5fps），这在终端里足够。但 Desktop 端的 React 渲染器可以轻松做到 60fps。更高的帧率是否更好？还是"低帧率 Spinner"本身就传递了"这是一个后台任务，不需要盯着看"的信号？

## Task 4：代码检索增强——Repo Map 思路（3 测试）

> 本 Task 来自报告一的架构建议，与交互展示无关，但对 Agent 质量影响最大，因此一并纳入。

### 4.1 问题

RouteDev 的代码检索只有 Grep（关键词搜索）和 Glob（文件模式匹配）。报告指出这是四条技术路线中最初级的一条。在大型项目中，Agent 经常需要"先找到相关代码在哪，再决定改什么"——但目前 Agent 只能通过关键词搜索盲找，效率低、token 消耗高。

### 4.2 Repo Map 思路（来自 Aider）

Aider 的 Repo Map 是一个轻量但高效的方案：

1. 用 tree-sitter 解析整个仓库的源文件
2. 提取每个文件的**类和函数签名**（不是完整内容）
3. 将签名列表注入到系统提示词中，作为"代码地图"
4. 模型读地图后，选择需要完整读取的文件

这比暴力搜索整个仓库省 token，比关键词搜索更精准（因为签名包含了代码结构信息）。

### 4.3 RouteDev 的实现思路

不需要从零实现 tree-sitter 解析器。RouteDev 已有 `code_search` 工具和 CodeGraph MCP 集成。可以考虑：

- **方案 A**：新建一个 `repo_map` 工具，用 TypeScript 的 Compiler API（`ts.createProgram`）解析项目结构，提取 export/import 关系和函数签名。这对 TypeScript 项目精准，但只支持 TS。
- **方案 B**：利用已有的 CodeGraph MCP 集成。CodeGraph 已经能做 `codegraph_explore` 和 `codegraph_search`——问题是它依赖外部 MCP 服务器，不是所有用户都会配置。
- **方案 C**：做一个轻量版的"文件结构扫描"——扫描目录树，列出每个 .ts 文件的 export 名称（用正则提取，不用 tree-sitter）。精度低但零依赖。

**需要你决定的：**
- RouteDev 主要面向什么语言的项目？如果主要是 TypeScript/JavaScript，方案 A 值得投入。如果是多语言，方案 C 更通用。
- Repo Map 的输出大小如何控制？一个 200 文件的项目，签名列表可能有 2000+ 行。需要有"按需加载"或"按相关度筛选"的机制，否则注入上下文反而浪费 token。
- 报告提到 Sourcegraph Cody 放弃了向量嵌入方案（对大仓库扩展性差），改用 BM25 排名 + 原生搜索。RouteDev 是否需要考虑 BM25 排名？还是简单的关键词搜索 + Repo Map 就够用？

## Task 5：过程评测指标（Harness-level Evaluation）（2 测试）

> 本 Task 来自报告一的架构建议。

### 5.1 问题

RouteDev 的 Phase 32 已经建立了分类器准确率测试（黄金测试集），但只测"分类对不对"。报告指出成熟的 Agent 评测应该同时度量过程指标：

- **单 trajectory 成本**：完成一个任务消耗了多少 token / 多少钱
- **工具调用次数**：完成一个任务调用了多少次工具
- **LLM 调用次数**：完成一个任务调用了几次模型
- **重试次数**：失败后重试了几次
- **首次成功率**：不做任何重试就成功的比例

### 5.2 RouteDev 的实现思路

TraceCollector 已有 7 种 span 类型（`react_iteration`/`tool_call`/`llm_call`/`worker_task`/`goal_step`/`compose_phase`/`hook`）。在此基础上，增加一个"trajectory 级"汇总：

- 每次任务完成时，从 TraceCollector 的 span 列表中计算汇总指标
- 将汇总写入 AuditLogger 的 JSONL（新增一种 action type：`trajectory_summary`）
- 字段包括：`taskId`、`totalTokens`、`totalCost`、`toolCallCount`、`llmCallCount`、`retryCount`、`durationMs`、`success`

这不需要复杂的 UI——数据存在 JSONL 里，后续可以用脚本分析。但它为未来的"Agent 质量看板"提供了数据基础。

**需要你思考的：** 汇总的触发点在哪？任务成功完成时记录是自然的。但任务失败、超时、用户取消时也应该记录——否则只统计成功案例会引入幸存者偏差。

## AGENTS.md 陷阱（预判）

| # | 陷阱 |
|---|------|
| 45 | `outputStyle` 替换 `disclosureLevel` 时，需要处理现有用户配置文件的向后兼容。建议在 schema.ts 中用 `z.preprocess` 做映射：`1 → minimal`，`2 → standard`，`3 → verbose` |
| 46 | 微摘要中的"关键决策"提取依赖 LLM 遵守 `<decision>` 标签格式。不是所有模型都会严格遵守——需要在解析时做容错处理（找不到 `<decision>` 时优雅降级为纯统计摘要） |
| 47 | Repo Map 的签名提取不能阻塞 Agent Loop 的主线程。如果仓库很大（500+ 文件），解析耗时可能超过 1 秒。考虑在 `app-init` 时预构建 Repo Map 并缓存，文件变更时增量更新 |
| 48 | trajectory 汇总的 span 列表可能包含跨任务的 span（如 `on-session-start` hook）。汇总时需要按 taskId 过滤，不要把不相关的 span 计入 |

## 不在本 Phase 范围内

| 项目 | 理由 |
|------|------|
| OS 级沙箱（Seatbelt/Bubblewrap） | 报告推荐企业级场景，RouteDev 7 层应用级安全够用 |
| Spec-Driven Development 框架 | 报告承认 SDD 仍处早期，RouteDev 的 Phase 文档模式已是轻量替代 |
| Background Agents（异步云端执行） | 需要云基础设施（VM/worktree 管理），超出桌面工具范围 |
| 向量嵌入语义搜索 | Sourcegraph Cody 已验证此方案对大仓库扩展性差，不建议跟进 |
| 多 Agent Token 传递优化 | 报告建议"选择性上下文传递"，但 RouteDev 的 WorkerExecutor 尚未在生产路径中充分使用，优化为时过早 |
