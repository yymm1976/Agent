# Phase 28 — 质量验收与发布准备

> **Phase 类型：** 质量门禁 + 发布准备（Quality Gates + Release Readiness）
> **前置依赖：** Phase 27 完成（v1.4.0，功能闭环）
> **目标版本：** v2.0.0
> **核心目标：** 通过全面的质量验收——测试覆盖、性能基线、安全审计、文档完整度、蓝图合规度——确认 RouteDev 达到商业交付标准，发布 v2.0.0

---

## 背景

Phase 26 清零了技术债务，Phase 27 补全了所有功能缺口。RouteDev 现在是功能完整、架构健康的产品。Phase 28 是最后的"验收考试"——不做新功能，只验证和加固。

本 Phase 的质量门禁标准参照商业软件交付惯例：
- **零 Critical / High 缺陷**
- **全量测试通过率 100%**
- **性能基线强制通过**（不通过则构建失败）
- **安全审计报告零未修复漏洞**
- **文档与代码完全同步**
- **蓝图合规度 ≥ 95%**

---

## Task 1：端到端用户旅程测试

**目标**：编写覆盖主要用户场景的 E2E 测试，确保从用户视角产品功能闭环。

### 测试场景清单

| # | 场景 | 验证要点 | 预期步骤数 |
|---|------|---------|-----------|
| E2E-1 | 新用户首次启动 | 配置向导 → 生成 config.yaml → 启动成功 | 3-5 |
| E2E-2 | 简单问答对话 | 用户输入 → 路由分类(simple) → LLM 调用 → 响应渲染 | 4-6 |
| E2E-3 | /goal 多步任务 | /goal 输入 → 计划分解 → 用户确认 → 逐步执行 → 验证通过 | 10-15 |
| E2E-4 | 权限确认流程 | shell_exec → PermissionEngine confirm → ConfirmDialog → 用户确认/拒绝 | 5-8 |
| E2E-5 | 分支对话 | 编辑消息 → 新分支创建 → 切换分支 → 验证消息历史 | 6-8 |
| E2E-6 | 检查点回滚 | 执行步骤 → 自动检查点 → /rollback → 验证文件恢复 | 8-10 |
| E2E-7 | Compose 管线全流程 | /compose → requirements → coding → testing → review → 完成 | 15-20 |
| E2E-8 | DurableExecutor 断点恢复 | /goal --durable → 执行中中断 → /resume → 从断点继续 | 10-12 |
| E2E-9 | MCP 工具调用 | MCP Server 连接 → 工具发现 → 工具调用 → 结果返回 | 6-8 |
| E2E-10 | 渠道集成消息流 | Telegram 消息 → 适配器解析 → Agent 处理 → 响应格式化 → 回复 | 8-10 |

### 实现方式

使用 Vitest + 模拟 LLM client（返回预设响应），验证完整链路从输入到输出。不依赖真实 LLM API。

### 验收

- 10 个 E2E 场景全部通过
- 新增 ≥ 10 个测试

## Task 2：性能基线强制门

**目标**：将性能测试从"记录结果"升级为"构建失败门"——任何性能退化都阻止发布。

### 性能基线（来自 Phase 23 performance-benchmark.test.ts）

| 指标 | 阈值 | 当前值（Phase 23） | 行动 |
|------|------|-------------------|------|
| 场景分类延迟 | < 1000ms | ~200ms | 添加强制断言 |
| 路由选择延迟 | < 100ms | ~20ms | 添加强制断言 |
| 端到端简单任务 | < 5000ms | ~2000ms | 添加强制断言 |
| Token 估算精度 | 误差 < 30% | ~15% | 添加强制断言 |
| 10x 连续调用稳定性 | 无 OOM / 崩溃 | 通过 | 添加强制断言 |
| 内存占用 | < 256MB | ~120MB | 新增断言 |
| 启动时间 | < 3000ms | ~1500ms | 新增断言 |

### 实现

1. 修改 `performance-benchmark.test.ts`，将所有 `console.log` 结果改为 `expect().toBeLessThan()` 断言
2. 在 `vitest.config.ts` 中将性能测试标记为 `testTimeout: 30000`
3. 新增 `scripts/perf-gate.ts`：独立运行性能门，返回 0/1 退出码
4. `package.json` 新增 `pnpm perf:gate` 脚本

### 验收

- `pnpm perf:gate` 全部通过
- 故意引入性能退化后构建失败
- 新增 ≥ 3 个测试（内存、启动时间、连续稳定性）

## Task 3：安全终审

**目标**：对 v1.4.0 代码库进行最终安全审计，确认零未修复漏洞。

### 审计清单

| # | 审计项 | 检查方法 | 通过标准 |
|---|--------|---------|---------|
| S-1 | 路径遍历 | 搜索所有文件操作，确认全部有路径校验 | 零未校验路径 |
| S-2 | 命令注入 | 搜索所有 `spawn`/`exec` 调用，确认参数已转义 | 零未转义参数 |
| S-3 | 凭据泄露 | 搜索 API key / secret / password 在代码和日志中的出现 | 零明文凭据 |
| S-4 | 权限绕过 | 模拟各种工具调用，确认 PermissionEngine 无绕过路径 | 零绕过 |
| S-5 | DoS 防护 | 验证 WebhookServer 限流、body 限制、认证 | 全部生效 |
| S-6 | 敏感文件保护 | 尝试读取 .env / credentials.json，确认被阻止 | 全部被阻止 |
| S-7 | 网络确认 | 尝试发起网络请求，确认需用户确认 | 确认对话框出现 |
| S-8 | 子进程隔离 | 确认 shell_exec 在子进程中执行，不污染主进程 | 主进程干净 |
| S-9 | 审计日志完整性 | 确认所有权限确认、工具执行、路由决策都有审计记录 | 无遗漏 |

### 产出

一份安全审计报告（`docs/SECURITY_AUDIT_v2.0.md`），列出每项审计的结果和证据。

### 验收

- 9 项审计全部 PASS
- 审计报告生成
- 新增 ≥ 5 个安全测试

## Task 6：蓝图合规度终审

**目标**：对照 BLUEPRINT.md 逐条检查实现状态，确保合规度 ≥ 95%。

### 合规度计算

```
合规度 = (已实现的蓝图"必须"功能数) / (蓝图"必须"功能总数) × 100%
```

### 审计方法

1. 逐条对照 BLUEPRINT.md 的每个"必须"功能
2. 对每个功能标记：✅ 已实现 / ⚠️ 部分实现 / ❌ 未实现
3. "部分实现"算 0.5 分
4. 列出所有未实现项及其理由（如"MVP 排除"、"设计变更"、"延期到 v3.0"）

### 产出

一份蓝图合规度报告（嵌入本 Phase 的执行报告中），包含：
- 逐项检查结果
- 合规度百分比
- 未实现项清单及理由

### 验收

- 蓝图"必须"功能合规度 ≥ 95%
- 所有未实现项有明确理由且标记为"非 MVP"或"延期"

经过 28 个 Phase 的迭代开发，RouteDev 正式发布 v2.0.0。

- ~1200+ 测试用例
- ~100 个源文件
- 零 Critical / High 缺陷
