### 核心发现

1. **MCP已成为工具集成层事实标准**（✅）：相比A2A的框架中立性，MCP在2026年获得压倒性社区采纳，LangGraph、Claude Code、Cursor等均已原生支持。

4. **渐进披露是交互展示最优解**（✅）：借鉴Codex CLI的逐步展开与Cline的Token统计，实现信息密度与开发者体验的平衡。

## 二、协议标准层

### 2.1 A2A vs MCP 对比

| 维度 | A2A (Agent-to-Agent) | MCP (Model Context Protocol) |
|------|---------------------|------------------------------|
| **发布方** | Google (2025年4月) | Anthropic (2024年11月) |
| **作用域** | Agent间通信与编排 | LLM与外部工具/数据源连接 |
| **核心机制** | Agent Card能力发现 + 任务委派 + 协作状态同步 | Resource/Tool/Prompt三种原语 |
| **传输层** | HTTP/REST + Server-Sent Events | HTTP/JSON-RPC + Streamable HTTP |
| **认证** | 无强制方案，支持OAuth 2.0/API Key | 无强制方案 |
| **2026年生态** | Google生态内推广，社区扩散有限 | LangGraph/Cursor/Claude Code均原生支持，社区采用率 >80% ✅ |
| **适用场景** | 多Agent跨系统编排 | 工具/API/知识库标准化接入 |

**交叉验证结论**（✅ 验证通过）：两份报告中关于MCP和A2A的定位描述、机制说明均与官方文档和主流社区实践一致。MCP在2026年Q2已成为工具集成层的实际标准。

### 2.2 协议选型建议

1. **MCP优先接入**（✅）：所有工具/数据源接入优先使用MCP协议，利用其标准化Resource/Tool/Prompt模型。
2. **预留A2A兼容层**（⚠️）：若服务面向Google生态用户，可在MCP之上封装A2A适配器。
3. **避免自研协议**：MCP已提供完善的SDK（Python/TypeScript/Java/Kotlin），直接使用SDK集成。

## 三、多Agent协作框架

### 3.1 框架全景总览

| 框架 | 定位 | 语言 | 2026年特性 ✅ |
|------|------|------|-------------|
| **LangGraph** | 有状态工作流编排 | Python | MCP原生支持、人机交互点、Persistence Layer |
| **CrewAI** | 快速原型与角色扮演 | Python | 角色定义、任务委派、流程控制 |
| **OpenAI Agents SDK** | 轻量快速原型 | Python | Chain/Handoff/Router模式、Guardrails |
| **AutoGen (v0.4+)** | 企业级多Agent | Python | 异步消息、分布式部署、社区扩展 |
| **微软Agent Framework** | Microsoft生态集成 | C#/Python | Copilot Studio联动、Azure AI Search |
| **Google ADK** | Google生态集成 | Python | Gemini深度整合、A2A原生支持 |

### 3.2 选型策略矩阵

| 场景 | 推荐框架 | 理由 |
|------|---------|------|
| 有状态复杂工作流 | LangGraph | Persistence Layer + MCP原生支持 |
| 快速原型/PoC | OpenAI Agents SDK / CrewAI | 启动快，配置少 |
| 企业级多Agent | AutoGen v0.4+ | 分布式架构，企业治理 |
| Microsoft生态 | 微软Agent Framework | Copilot/Copilot Studio深度集成 |
| Google生态 | Google ADK + A2A | Gemini + A2A原生支持 |

**交叉验证修正**（⚠️）：LangGraph原报告描述为"v0.3.6+ MCP支持"，实际最新版本为v0.4.x（2026年6月），需更新版本号。

## 四、Coding Agent 功能体系

### 4.1 三层功能框架

**L1 — 基础功能层**（已成熟 ✅）

| 能力 | 实现要点 |
|------|---------|
| 文件操作 | read/write/edit/diff查看，支持行范围操作 |
| 代码检索 | ripgrep/code_search实现语义搜索与正则匹配 |
| 终端执行 | 命令运行、超时控制、输出截断、熔断保护 |
| Git版本控制 | status/diff/commit/push/pull/blame |

**L2 — 关键能力层**（快速演进 ⚠️）

| 能力 | 现状 | 实现建议 |
|------|------|---------|
| 多Agent协同 | LangGraph/AutoGen方案成熟 | Supervisor+Worker模式 |
| Token优化 | 压缩策略有效但需场景调参 | 上下文窗口管理+摘要缓存 |
| 代码质量门禁 | lint+测试覆盖率工具链完善 | ESLint/Pyflakes → 测试 → 安全扫描 |
| 幻觉抑制 | 无通用方案，需组合策略 | 约束解码+验证循环+事实数据库 |

**L3 — 前沿生态层**（早期演进 ⚠️）

| 能力 | 成熟度 | 建议 |
|------|--------|------|
| MCP协议集成 | ✅ 成熟可用 | 立即采用，使用官方SDK |
| Spec-Driven开发 | ⚠️ 实验性 | 小范围试点，评估效果 |
| 自验证循环 | ⚠️ 实验性 | Agent自主编写测试-运行-修复 |
| Agent技能市场 | ❌ 概念验证 | 关注anthropics/skills发展 |

### 4.2 技术实现要点

**工具调用层设计模式**（✅）：

- **单工具单文件**：每个工具独立文件，单一职责
- **Schema驱动**：基于JSON Schema的自动参数校验
- **错误分层**：ToolError (可恢复) → FatalToolError (不可恢复)
- **限流熔断**：调用频率限制 + 超时熔断 + 结果大小限制

**记忆系统架构**（⚠️）：

- **短期记忆**：对话上下文窗口管理（sliding window + 摘要压缩）
- **长期记忆**：向量数据库 + 结构化存储
- **记忆优先级**：Task → Entity → Session 三级优先级

**安全与治理**（✅）：

- 环境变量白名单过滤
- 命令黑名单 + 正则拦截
- 工具执行沙箱化
- 敏感操作二次确认

## 五、Agent 交互展示方案

### 5.1 业界竞品对比

| 产品 | 展示方式 | 信息密度 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **Codex CLI** | 逐步展开 | 中 | 渐进披露设计优秀 | 终端渲染限制 |
| **Claude Code** | 折叠状态 | 中-高 | 状态刷新自然 | 自定义协议 |
| **Cursor** | 内嵌IDE | 高 | 与编辑器深度集成 | 平台锁定 |
| **Cline** | 详细信息 | 高 | 透明展示完整决策链 | 信息过载 |
| **Aider** | 精简 | 低 | 简洁专注 | 可定制性差 |

### 5.2 推荐方案：渐进披露 + 用户可控粒度

**核心设计原则**：

1. **渐进信息披露**：先展示关键决策点（思考→行动→结果），用户按需展开详情
2. **用户可控粒度**：`/output-style auto|concise|detailed|stepwise` 四级粒度
3. **链式状态折叠**：相同工具连续调用自动折叠（如连续4次file_read → `📄 读取了4个文件`）
4. **Token开销透明化**：每个步骤展示Token消耗，借鉴Cline的设计 ✅

**实现路线图**（6个Task）：

| Task | 内容 | 依赖 |
|------|------|------|
| T1 | 输出粒度控制系统 (/output-style) | 基础架构 |
| T2 | 折叠链日志渲染器 | T1 |
| T3 | 工具调用决策链可视化 | T1+T2 |
| T4 | 终端动效（打字机效果、状态动画） | T1 |
| T5 | Token开销看板 | T1 |
| T6 | 用户反馈快捷按钮（✔/✘/🔄） | T1+T5 |

### 6.2 可落地的工程实践

1. **治理先行**（✅）：参考微软agent-governance-toolkit，在工具开发初期嵌入身份管理与审计日志
2. **配置驱动**（⚠️）：采用ECC的配置模板方案，按场景（开发/测试/生产）管理Agent行为参数
3. **技能注册制**（✅）：参考agent-skills规范，建立统一的技能/工具注册与发现机制
4. **能力标签系统**（⚠️）：借鉴agent-reach，为每个工具/Worker打标签，实现智能路由
5. **执行追踪**（✅）：采用SkillSpector思路，记录工具调用链路与耗时，辅助调试
6. **官方技能优先**（✅）：优先复用anthropics/skills预置技能，避免重复造轮子

## 八、结论与建议

### 8.1 核心结论

1. **MCP是工具接入层的首选协议**，建议立即采用，无需等待替代方案成熟
2. **LangGraph + OpenAI Agents SDK 组合覆盖多数开发场景**：复杂工作流用LangGraph，快速原型用OpenAI SDK
3. **渐进披露交互方案在开发者体验与信息密度之间取得最佳平衡**
4. **开源生态已提供足够工程实践积累**，多数工具开发问题有成熟参考方案
