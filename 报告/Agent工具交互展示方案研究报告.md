## 一、业界竞品调研

### 1.1 Codex CLI（OpenAI）

**架构：** Rust 重写版，终端 Agent。ReAct 循环：Prompt 构造 → 模型推理 → 工具调用 → 反馈整合 → 循环迭代。

**展示方式（实际观察）：**
- 使用**简洁的动词化输出**：`Applied`（已应用编辑）、`Wrote`（已写入文件）、`Patched`（已修补）、`Ran`（已运行命令）
- **不展示推理过程**——模型内部的思考（Chain of Thought）对用户完全隐藏
- 完成后展示**精简的变更摘要**：哪个文件、改了哪里、结果如何
- 使用 `/plan` 模式先规划后执行，规划阶段展示完整计划，执行阶段收起细节
- 沙盒模式决定展示密度：`read-only` 模式下静默执行，`danger-full-access` 模式下强制确认

**关键设计决策：**
- 思考过程完全不可见（与 Claude Code 相反）
- 工具调用仅展示结果，不展示参数细节
- 使用过去时动词（`Applied`、`Wrote`）而非进行时，强调"已完成"
- 所有输出集中在**结果**，而非**过程**

**对轻度用户：** 极简，几乎看不到任何过程信息
**对重度用户：** 可通过 `--verbose` 或 tracing 查看完整日志；但日常使用中信息密度低

---

### 1.2 Claude Code（Anthropic）

**架构：** Node.js，Thought → Action → Observation 状态机。强制模型在工具调用前输出 `<thought>` 标签。

**展示方式（源码分析）：**
- 推理过程使用 `<thought>` 标签包裹，**默认折叠**但可展开
- 工具调用展示为**一行摘要**：工具名 + 关键参数（如文件路径）
- 执行结果展示为**一行结果**：成功/失败 + 关键数据
- 提供 `/output-style` 命令，用户可自定义输出格式
- 权限控制决定展示密度：只读操作静默，可逆写操作询问，不可逆操作强制确认

**关键设计决策：**
- 思考过程存在但默认隐藏——用户想看可以展开
- 使用 `<thought>` 语义标签而非纯文本，便于前端解析和折叠
- Output Style 机制让用户（而非开发者）决定信息密度
- 失败时自动展开 `<thought>`，让用户看到模型的推理过程

**对轻度用户：** 接近 Codex，思考折叠、动作摘要
**对重度用户：** 可展开 `<thought>`、查看完整工具调用链、自定义 output style

---

### 1.3 Cursor（IDE 集成）

**架构：** VS Code 插件，Ask / Plan / Agent 三种模式。

**展示方式：**
- **Agent 模式**：展示步骤列表，每步有图标 + 状态描述
- 编辑时显示 `Applying...` 动画，完成后展示 diff 预览
- 终端命令显示 `Running: npm test` 并实时输出
- 使用绿色/红色高亮标记代码变更（类似 Git diff）

**对轻度用户：** 界面友好，可视化 diff，但 IDE 内信息密度天然较高
**对重度用户：** 可查看完整工具调用日志、终端输出、每一步的详细信息

---

### 1.4 竞品共性与差异总结

| 维度 | Codex CLI | Claude Code | Cursor |
|------|-----------|-------------|--------|
| 思考过程 | 完全隐藏 | 默认折叠 | 不展示 |
| 工具调用 | 过去时动词 + 结果 | 一行摘要 | 步骤列表 |
| 结果展示 | 变更摘要 | 成功/失败 + 数据 | Diff 视图 |
| 信息密度 | 极低 | 可配置 | 中等 |
| 用户控制 | 沙盒模式 | /output-style | 模式切换 |
| 动效 | 无 | 无 | 进度条/spinner |
| 失败处理 | 错误信息 | 自动展开 thought | 错误高亮 |

**核心发现：**
- **没有工具使用复杂的动画或 Spinner 系统**——它们在终端里用"简洁文本"代替了"动效"
- 所有工具都在做**信息减法**，而非加法——默认展示越少越好
- 差异化的关键在于**用户控制权**——让用户决定看到多少，而非开发者预设

## 二、已有方案评估

### 2.1 方案 A：本对话中提出的"渐进披露 + 动效"方案

**核心内容：**
- L1/L2/L3 三层渐进披露
- 固定图标 + 语义色 + 动态文本
- Spinner 动画 + 进度条 + 时间计数器
- 心跳机制（30s/90s 阈值检测）
- 完成后折叠 + 微摘要
- 可配置的折叠策略

**优点：**
- 信息分层清晰，用户可按需获取
- 动效消除"卡死焦虑"
- 完成后折叠符合注意力管理
- 配置化设计兼顾不同用户

**缺点：**
- 终端 Spinner 动画价值有限（竞品都没用）
- 三层渐进披露在终端里实现复杂（需要键盘交互切换层级）
- 心跳机制在 LLM 流式场景下难以精确判断（收到 token ≠ 任务正常）
- 图标系统过度设计——终端不是 Web，图标多反而干扰阅读
- 部分设计来自 Web/IDE 场景的惯性思维，直接搬到终端可能水土不服

**判定：** 方案完整但过度设计。剥离动效层后，核心的渐进披露 + 完成后折叠是正确方向。

---

### 2.2 方案 B：Gemini 建议的"可折叠手风琴 + 时间轴"

**核心内容：**
- 可折叠的思考手风琴（执行中展开，完成后收起）
- 非对称时间轴（每步一行）
- 动态文本（根据阶段变化）
- Diff 视图而非完整代码
- 完成后淡化历史步骤

**优点：**
- 手风琴模式直观，执行中展开、完成后收起符合直觉
- 时间轴比纯文本更结构化
- 淡化历史步骤是优秀的设计
- 按用户类型（零基础/普通/高级）调整密度是正确方向

**缺点：**
- "手风琴"在终端里实现困难（Ink 做不了 CSS 动画）
- 时间轴在终端中受限于行宽，复杂任务可能显示不全
- 未涉及"动效"层——用户仍然不知道 Agent 是否卡死
- 缺少对"静默回退"的展示策略

**判定：** 交互逻辑正确，但部分 UI 模式来自 Web 思维，需要翻译为终端原生方案。

---

### 2.3 方案 C：用户补充的"完成后全部折叠"

**核心内容：** 任务完成后所有过程折叠，只保留最终输出。

**优点：**
- 最大化信噪比
- 符合注意力管理（U 型曲线）
- 实现简单，不需要复杂的状态管理

**缺点：**
- 失败时也应该折叠吗？（不应该）
- 折叠后用户如何复盘？（需要"展开"入口）
- 完全折叠可能让用户怀疑"AI 到底有没有干活"

**判定：** 方向正确，但需要补充"微摘要"和"失败时不折叠"的差异化策略。

## 三、方案判定与排名

### 3.1 各方案逐维度评分

| 维度 | 方案 A（渐进+动效） | 方案 B（手风琴+时间轴） | 方案 C（完成后折叠） | Codex 做法 | Claude 做法 |
|------|:---:|:---:|:---:|:---:|:---:|
| 信息密度控制 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 终端适配性 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 轻度用户友好 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 重度用户友好 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 消除卡死焦虑 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐ | ⭐ | ⭐ |
| 实现复杂度 | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 可配置性 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ |

### 3.2 核心判定

1. **竞品（Codex、Claude Code）不依赖复杂动效，但它们的"简洁"是有代价的**——用户确实会焦虑"AI 是不是卡死了"。Codex 和 Claude Code 的用户主要是专业开发者，他们对"终端卡住"有心理预期。如果你的目标用户包括轻度用户，动效层是必要的。

2. **Claude Code 的 `/output-style` 是最值得借鉴的设计**——它把"信息密度"交给用户控制，而非开发者预设。这是"兼顾轻度/重度用户"的最佳实践。

3. **方案 A 的动效层过度设计，但核心思想（渐进披露、心跳、完成后折叠）是正确的**。需要做减法而非加法。

4. **方案 B 的"手风琴"和"时间轴"在终端里需要翻译为更原生实现**——用 Ink 的 `<Box>` 和 `<Text>` 组件的条件渲染，而非 CSS 动画。

5. **方案 C 的"完成后折叠"是终端场景下的最佳决策**，但需要补充"微摘要"和"失败不折叠"。

## 四、最终方案设计

### 4.1 设计原则

| 原则 | 含义 |
|------|------|
| **终端原生** | 不使用 Web/IDE 思维的设计模式（手风琴动画、CSS transition），使用 Ink 条件渲染 + 文本切换 |
| **默认极简** | 轻度用户看到的最少——当前动作 + 结果 |
| **按需深入** | 重度用户通过配置或快捷键获取完整信息 |
| **用户掌权** | Output Style 配置决定信息密度，而非开发者预设 |
| **失败优先** | 失败时自动展开，成功时自动折叠——不对称设计 |

### 4.2 用户分层策略

```
┌─────────────────────────────────────────────────────┐
│                    用户分层                           │
│                                                     │
│  轻度用户 (output-style: minimal)                   │
│  ├─ 思考过程：完全隐藏                               │
│  ├─ 工具调用：仅展示结果（动词过去时）               │
│  ├─ 进度：仅展示当前步骤目标                         │
│  ├─ 动效：无                                        │
│  └─ 完成后：立即折叠，只留结果                       │
│                                                     │
│  普通用户 (output-style: standard) [默认]            │
│  ├─ 思考过程：折叠但可展开                           │
│  ├─ 工具调用：一行摘要 + 状态图标                    │
│  ├─ 进度：进度条 + 步骤计数                          │
│  ├─ 动效：简洁 Spinner + 时间计数器                  │
│  └─ 完成后：折叠 + 微摘要（步骤数、耗时、关键决策）  │
│                                                     │
│  重度用户 (output-style: verbose)                   │
│  ├─ 思考过程：默认展开                               │
│  ├─ 工具调用：完整参数 + 返回结果                    │
│  ├─ 进度：完整时间轴 + Token 消耗                    │
│  ├─ 动效：全部展示                                   │
│  └─ 完成后：保留完整过程日志                         │
└─────────────────────────────────────────────────────┘
```

### 4.3 终端原生实现方案

#### 4.3.1 三层信息架构（翻译为终端版本）

```
L1（始终可见，1-2 行）：
┌─────────────────────────────────────────────────────┐
│ 模型: gpt-4o │ 模式: build │ 编排: coding 2/4       │
│ ▶ 正在优化 utils.py 的重复逻辑...                    │
└─────────────────────────────────────────────────────┘

L2（按 d 展开，3-8 行）：
  ✅ 已分析项目结构（2 个文件）
  ✅ 已读取 config.json（配置正常）
  📝 已优化 utils.py  (+12 / -3)  [查看 Diff]
  🔄 正在运行单元测试... (3s)

L3（再按 d，完整日志）：
  [14:32:05] tool_call: file_read  config.json  12ms
  [14:32:06] llm_call: gpt-4o  420→180 tokens  340ms
  [14:32:07] tool_call: file_write  utils.py  8ms
```

**关键翻译：**
- L1 = StatusBar 组件（已存在）
- L2 = 动作流列表（StepCard 组件，已存在）
- L3 = TracePanel 组件（已存在）
- 层级切换 = 键盘 `d` 键（已在 TracePanel 中实现）

#### 4.3.2 动效的极简实现

**竞品分析结论：Codex 和 Claude Code 不使用复杂动画。** 因此动效应该做减法：

| 场景 | 动效 | 理由 |
|------|------|------|
| 模型调用中 | 无（等待流式 token） | 竞品共识：等待 LLM 时不需要动画 |
| 工具执行中 | 简洁 Spinner（`⠋`） | 工具执行可能耗时较长，需要信号 |
| 长时间运行（>30s） | 时间计数器 | 消除"等了多久"的不确定性 |
| 可能卡住（>90s） | 计时器 + 取消提示 | 给用户退出选项 |

**决策：放弃复杂的 Spinner 动画系统，只保留两个动效元素：**
1. 工具执行时的简洁 Spinner（`⠋` 单字符，每 200ms 切换）
2. 超过 30s 时显示"已运行 XXs"计时器

#### 4.3.3 Output Style 配置

```yaml
# 全局配置（config.yaml）
ui:
  outputStyle: standard       # minimal | standard | verbose
  showThinking: false         # 是否展示思考过程
  showToolParams: false       # 是否展示工具调用参数
  showTokenUsage: false       # 是否展示 Token 消耗
  collapseOnComplete: true    # 完成后是否折叠
  autoCollapseDelayMs: 1500   # 折叠延迟
  expandOnError: true         # 失败时是否自动展开
  idleHintSeconds: 30         # 空闲提示触发秒数
  bell: true                  # critical 通知是否触发 bell
```

**灵感来源：** Claude Code 的 `/output-style` 命令。但 Claude Code 的 output-style 更偏向"改变回复风格"，本方案将其扩展为"控制信息密度"。

#### 4.3.4 动作动词体系

借鉴 Codex CLI 的动词过去时风格，但增加进行时（执行中）和完成时（执行后）的区分：

| 工具类型 | 执行中 | 执行后 | 失败 |
|---------|--------|--------|------|
| 读取文件 | `⠋ 读取 config.json...` | `已读取 config.json (4KB)` | `读取失败：权限不足` |
| 编辑文件 | `⠋ 修改 utils.py...` | `已修改 utils.py (+12/-3)` | `修改失败：文件被锁定` |
| 执行命令 | `⠋ 运行 npm test...` | `测试通过 (3/3)` | `测试失败 (2/3)` |
| 搜索代码 | `⠋ 搜索认证逻辑...` | `找到 5 个匹配` | `搜索无结果` |
| 创建文件 | `⠋ 创建 auth.ts...` | `已创建 auth.ts` | `创建失败：磁盘空间不足` |

**设计原则：**
- 执行中：`Spinner + 动词进行时 + 目标`
- 执行后：`动词过去时 + 目标 + 关键数据`
- 失败：`动词过去时 + 失败原因 + 建议操作`

#### 4.3.5 完成后折叠策略

```
✅ 成功 → 折叠 + 微摘要
❌ 失败 → 展开 + 高亮错误 + 操作按钮
⏸️ 暂停 → 不折叠，等待用户决策
```

**微摘要内容（成功时）：**
```
✅ 任务完成：utils.py 性能优化
─────────────────────────────────────────
📊 摘要：4 步 / 2m15s / +12 -3 / 测试通过

💡 关键决策：
  • 将 O(n²) 循环改为 Map 查找（性能提升 ~10x）
  
[展开执行过程]  [查看 Diff]  [运行 /test]
─────────────────────────────────────────
```

**关键决策的提取：** 在系统提示词中要求 LLM 在完成任务时用 `<decision>` 标签标记关键决策，前端解析后展示在微摘要中。

## 五、实现路线图

### Phase 26：Output Style 系统

**目标：** 实现 `output-style` 配置化，让轻度/重度用户获得不同信息密度。

**Task 1：Output Style 配置**
- 扩展 `src/config/schema.ts`：新增 `OutputStyleSchema`（`minimal | standard | verbose`）
- 扩展 `src/config/defaults.ts`：默认 `standard`
- 新增 `/output-style` 命令，支持运行时切换

**Task 2：StepCard 按 outputStyle 条件渲染**
- `minimal`：只显示完成/失败状态，不显示进行中步骤
- `standard`：显示当前步骤 + 已完成的摘要
- `verbose`：显示所有步骤详情

**Task 3：TracePanel 按 outputStyle 控制默认层级**
- `minimal`：默认 L1（仅摘要）
- `standard`：默认 L2（关键细节）
- `verbose`：默认 L3（完整日志）

**Task 4：新增 HeartbeatBar 组件**
- 极简实现：Spinner（`⠋`）+ 计时器（超过 30s 时显示）
- 超过 90s 显示取消提示
- 仅在 `standard` 和 `verbose` 模式下启用

**Task 5：微摘要增强**
- 在系统提示词中增加 `<decision>` 标签要求
- `renderGoalCompletionSummary` 中解析并展示关键决策

**Task 6：测试与验证**
- 新增测试 ≥ 20 个
