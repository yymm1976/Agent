## 一、项目概览

RouteDev 是一款 AI 驱动的 CLI + 桌面开发助手，核心能力是根据任务复杂度（simple/medium/complex/reasoning）自动路由到不同的 LLM 模型。项目同时提供 CLI 终端界面（Ink + React）和 Electron 桌面 GUI 两种交互形态。历经 52 个 Phase 的迭代开发，当前版本 v4.0.1。

**技术栈**：TypeScript 6.x (strict, ESM) / Node.js 20+ / pnpm 11+ / Ink 7 + React 19 (CLI) / Electron 30 + electron-vite (桌面) / Vite 6 + Tailwind 3 + Zustand + Recharts (桌面渲染) / Vitest 4 (测试) / tsup 8 (CLI 构建) / @anthropic-ai-sdk + openai (LLM) / zod + winston + simple-git + yaml + chalk

### 4.2 关键架构发现

**发现 1：setter 注入缺口（高优先级）**

`ReActAgentLoop` 仅有 8 个 setter，但 `app-init.ts` 创建了远超 8 种功能模块。如果策略引擎、引用系统、宏系统、调度引擎等模块在 app-init.ts 中被实例化后，因 Loop 缺少对应 setter 而无法注入，则这些模块实际上是"静默失效"的。这是本项目最大的架构隐患——大量精心设计的模块可能在运行时完全不起作用。

**建议**：逐一检查 app-init.ts 中每个实例化对象的后续 setter 调用链，确认是否存在"创建后丢弃"的模式。对于确认断开的模块，要么在 Loop 中添加对应 setter，要么将模块标记为计划删除。

**发现 2：动态导入 + fail-open 模式的普遍化**

较新的模块（persona-engine、voice-manager、eq-detector、parallel-experiment、market-manager、quality-gate、registry-client、skill-flow-engine、dual-loop-orchestrator）统一使用 `app-init.ts` 中的动态导入 + try-catch fail-open 模式。这意味着这些模块如果在运行时加载失败，不会阻断应用启动，但也不会产生任何日志警告（仅有 debug 级别日志）。

**建议**：对 fail-open 的模块添加 `logger.warn()` 级别的失败通知，让用户知道某些功能未能加载。

**发现 3："经过测试的死代码"**

skills/ 目录中 9 个死代码文件中有 7 个拥有完善的单元测试。这说明开发流程中测试纪律良好，但缺乏"装配验证"环节——即测试通过了但模块从未被接入生产路径。

**建议**：添加集成测试或装配验证脚本，确保所有模块至少被一条生产路径引用。
